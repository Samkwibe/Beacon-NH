BeaconNH deployment checklist
==============================

Frontend (Vercel, Netlify, etc.)
--------------------------------
1. Build: npm install && npm run build (output: dist/).
2. Set environment variables at build time (Vite reads VITE_* only while building):
   VITE_FIREBASE_* — same keys as .env.example at repo root
   VITE_API_URL — public HTTPS URL of your API, e.g. https://api.yourdomain.com
   (no trailing slash)
   VITE_PUBLIC_ADMIN — set to "true" only if this deploy should expose the admin UI (default:
   omitted = admin route not included in the production bundle’s router, so /admin is not public).
   VITE_ADMIN_ROUTE — optional single path segment to replace the default "admin" URL with a
   harder-to-guess segment (e.g. ops-7f3a9c2e).
3. vercel.json provides SPA fallback for React Router.
4. Netlify: public/_redirects is copied into dist/ so client-side routes work.

Backend API (Node server/)
---------------------------
Required env vars:
  MYSQL_HOST, MYSQL_PORT, MYSQL_USER, MYSQL_PASSWORD, MYSQL_DATABASE
    Railway’s MySQL template exposes MYSQLHOST, MYSQLPORT, MYSQLUSER, MYSQLPASSWORD, MYSQLDATABASE;
    this API accepts either naming style.
  CORS_ORIGIN — origin(s) of your deployed SPA. One origin, or comma-separated, e.g.
    https://beacon-nh.vercel.app,https://www.yourdomain.org
    Omit or use carefully in production; reflect-origin (true) is convenient for local dev only.
  Firebase Admin (pick one):
    GOOGLE_APPLICATION_CREDENTIALS — path to service account JSON inside the container/host
    FIREBASE_SERVICE_ACCOUNT_JSON — full JSON string of the service account (many hosts prefer this)

Local stack: MySQL + API in Docker
-----------------------------------
1. Place Firebase service account JSON at server/firebase-admin.json (never commit it).
2. docker compose up -d mysql   — database only (API via npm run api:dev on host)
   OR
   npm run docker:stack          — mysql + API container (profile "api")
3. Edit docker-compose.yml api.service.environment.CORS_ORIGIN for your frontend URL.
4. With Docker API on host port 3002: set VITE_API_URL=http://localhost:3002 in root .env for Vite.
   Health check: GET http://localhost:3002/api/health
   Host-only API (npm run api:dev): http://localhost:3001/api/health

Railway (API + MySQL on Railway)
--------------------------------
Repo: https://github.com/YOU/BeaconNH (or your fork). You need one Railway **project** with two
services: **MySQL** and the **Beacon API**.

1) Create project
   - railway.app → New Project → Deploy from GitHub → select this repository.

2) Add MySQL
   - Project canvas → + New → Database → MySQL (or deploy from
     https://railway.com/template/mysql ).
   - Wait until MySQL is running. No schema yet.

3) Add the API service
   - + New → GitHub Repo → same repository again (or “Empty service” → connect repo).
   - **Build configuration** (pick one — if you skip all, Railpack runs `npm ci` on the **repo root**
     Vite app and can fail if the root `package-lock.json` drifts):
     - **A:** The repo includes **`railway.toml`** at the **repository root**; it sets
       `Dockerfile.api` so a service connected with default root directory builds the **API**
       with Docker (recommended for this monorepo).
     - **B:** Service → Settings → **Root Directory** → `server` (uses `server/Dockerfile`).
     - **C:** Leave Root Directory empty and set **Variables** → **`RAILWAY_DOCKERFILE_PATH`** =
       **`Dockerfile.api`** (overrides auto-detect if `railway.toml` is not picked up).
   - Your GitHub branch must include **`server/package-lock.json`** (API image) and a consistent
     root **`package-lock.json`** if Railpack ever builds the frontend on Railway.
   - Service → Settings → **Networking** → Generate Domain (public HTTPS URL). This is your API base
     (e.g. https://beacon-api-production.up.railway.app). No trailing slash.

4) Wire MySQL into the API
   - Open the **API** service → **Variables**.
   - Add **Variable Reference** (or “Reference”) from the **MySQL** service and include at least:
     MYSQLHOST, MYSQLPORT, MYSQLUSER, MYSQLPASSWORD, MYSQLDATABASE
     (Railway docs: https://docs.railway.com/databases/mysql — connect via references.)
   - The Node app reads MYSQLHOST / MYSQLUSER / … automatically.

5) API-only variables (same service → Variables)
   - CORS_ORIGIN — your Vercel origins, comma-separated if needed, e.g.
     https://beacon-nh.vercel.app
   - FIREBASE_SERVICE_ACCOUNT_JSON — paste the **entire** Firebase service account JSON (one line).
     Same Firebase **project** as VITE_FIREBASE_* on Vercel. Get it from Firebase Console → Project
     settings → Service accounts → Generate new private key.
   - Optional: MYSQL_SSL=true if connections fail (some network paths need TLS; try without first).

6) Initialize the database
   - Run server/schema.sql once against the MySQL database. Options:
     - Railway MySQL → **Data** / **Query** tab if available, paste the SQL; or
     - Local: install mysql client, use Railway MySQL **Connect** / public TCP proxy URL and credentials
       from the MySQL service variables; run: mysql -h HOST -P PORT -u USER -p < server/schema.sql

7) Deploy / verify
   - Trigger deploy on the API service. Logs should show the API listening.
   - Open GET https://YOUR-API-DOMAIN/api/health — expect JSON with "mysql": true and
     "firebaseAdmin": true.

8) Point Vercel at Railway
   - Vercel → Environment Variables → Production:
     VITE_API_URL=https://YOUR-API-DOMAIN   (no trailing slash)
     VITE_FIREBASE_* — match your Firebase web app
   - Redeploy Vercel so the build picks up VITE_API_URL.

Pricing note: Railway bills for MySQL + API runtime and egress; see railway.app/pricing.

Production MySQL
----------------
Use a managed database (RDS, PlanetScale, etc.). Point MYSQL_* at that instance.
Run migrations/schema from server/schema.sql once on the empty database.
If the provider requires TLS, set MYSQL_SSL=true on the API (see server/.env.example).

Full stack (example: Vercel site + hosted API + managed MySQL)
---------------------------------------------------------------
Goal: visitors load the SPA from Vercel; events and RSVPs go through your Node API and MySQL.

A) Database
   1. Create a MySQL 8 database (AWS RDS, PlanetScale, Aiven, DigitalOcean Managed DB, etc.).
   2. Run server/schema.sql once in a SQL console against the empty database.
   3. Note host, port, user, password, database name.

B) API (Node service — not Vercel static hosting)
   Deploy the server/ folder as a long-running web service (Dockerfile provided). Typical hosts:
   **Railway** (step-by-step above), Render, Fly.io, Google Cloud Run, or any VPS with Node 22+.
   Required service env vars:
     PORT — set by the host if provided (otherwise 3001).
     MYSQL_HOST, MYSQL_PORT, MYSQL_USER, MYSQL_PASSWORD, MYSQL_DATABASE
     CORS_ORIGIN — your live site origin(s), comma-separated, e.g.
       https://beacon-nh.vercel.app,https://your-custom-domain.com
       (no trailing slashes; must match the browser’s exact origin.)
     FIREBASE_SERVICE_ACCOUNT_JSON — paste the full Firebase service account JSON as a single line,
       OR use GOOGLE_APPLICATION_CREDENTIALS with a mounted file path inside the container.
   Health check: GET https://YOUR-API-HOST/api/health — expect mysql: true and firebaseAdmin: true.

C) Frontend (Vercel)
   In the Vercel project → Settings → Environment Variables (Production):
     VITE_API_URL=https://YOUR-API-HOST   (no trailing slash)
     VITE_FIREBASE_* — same Firebase web app as local .env.example
     VITE_PUBLIC_ADMIN=true — only if this deployment should include the admin UI route
   Redeploy after saving env vars so Vite embeds VITE_* into the build.

D) Firebase Auth (admin + API writes)
   - Enable Email/Password (or your chosen provider) in Firebase Console → Authentication.
   - Create at least one admin user for the /admin dashboard.
   - The API verifies Bearer tokens with Firebase Admin; without FIREBASE_SERVICE_ACCOUNT_JSON the
     admin and write endpoints return 503/401.

E) Smoke test
   - Open the production site → Events should load from GET /api/events.
   - Submit a test RSVP → row appears in MySQL; list RSVPs in admin after signing in.

Secrets
-------
Do not commit root .env, server/.env, or firebase-admin.json.
Rotate Firebase web API key if .env was ever pushed to a public repo.
